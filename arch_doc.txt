A. Application Overview
SmartSure is an enterprise-grade, microservices-based insurance management platform that digitizes the complete insurance lifecycle. It enables customers to register, purchase policies, upload documents, and file claims through guided workflows. Admins manage products, verify claims, and generate operational reports.

Tech Stack


B. High-Level Design (HLD)
System Architecture Diagram
All Angular UI traffic flows exclusively through the Ocelot API Gateway. The gateway validates JWT, enforces rate-limiting, and routes to the appropriate microservice. Services communicate asynchronously via RabbitMQ events. Each service owns its isolated SQL Server database.


Event-Driven Choreography Flow вҖ” Policy Purchase & Notifications
Services communicate purely through RabbitMQ events вҖ” no service calls another directly. Each service publishes an event when something meaningful happens; interested services listen and react independently. This is choreography: decoupled, asynchronous, and resilient. There is no central orchestrator coordinating the flow.

Key Design Decisions
Repository Pattern: Controller вҶ’ Service вҶ’ Repository вҶ’ DbContext in every microservice
Dependency Injection: All services, repositories, and infrastructure registered via IServiceCollection
Global Exception Handling: Custom middleware catches all unhandled exceptions, logs via Serilog, returns RFC 7807 ProblemDetails
IMemoryCache: Applied on policy types listing, insurance product catalog, and user role lookups (5-min TTL)
Polly in Ocelot: 3 retries with exponential backoff + circuit breaker (5 failures вҶ’ 30s open)
Serilog: Structured JSON logging to console + rolling file sink; includes CorrelationId per request

E. Database Design (SQL Server + EF Core Code-First)
Each microservice owns its isolated SQL Server database with its own DbContext and migrations. Cross-service references are stored as plain GUID columns вҖ” NO foreign key constraints cross service boundaries. Data consistency is maintained via RabbitMQ events.


DB 1 вҖ” SmartSure_AuthDb  (Auth Service :5001)
Owns all identity data: users, roles, and credentials. JWT is stateless вҖ” no refresh token table. IMemoryCache is used for user role lookups to avoid repeated DB hits on every authenticated request.

F. Cross-Cutting Implementation Concepts
1. Global Exception Middleware (SmartSure.Shared)
ExceptionMiddleware lives in SmartSure.Shared вҖ” written once, used by all 4 services via project reference. Each service's Program.cs registers it with a single line. Custom exception types also live in Shared so any service can throw them and the middleware catches them uniformly.
// SmartSure.Shared/Middleware/ExceptionMiddleware.cs
using SmartSure.Shared.Exceptions;

public class ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
{
  public async Task InvokeAsync(HttpContext ctx) {
    try { await next(ctx); }
    catch (NotFoundException ex)      { await Write(ctx, 404, ex.Message); }
    catch (ValidationException ex)    { await Write(ctx, 400, ex.Message); }
    catch (UnauthorizedException ex)  { await Write(ctx, 401, ex.Message); }
    catch (Exception ex) {
      logger.LogError(ex, "Unhandled exception [{CorrelationId}]", ctx.TraceIdentifier);
      await Write(ctx, 500, "An unexpected error occurred");
    }
  }
  // Returns RFC 7807 ProblemDetails вҖ” never raw stack traces
}

// SmartSure.Shared/Exceptions/NotFoundException.cs
public class NotFoundException(string message) : Exception(message);

// SmartSure.Shared/Exceptions/ValidationException.cs
public class ValidationException(string message) : Exception(message);

// Each service Program.cs вҖ” one line, no duplication
using SmartSure.Shared.Middleware;
app.UseMiddleware<ExceptionMiddleware>();

2. JWT Configuration
Issued by Auth Service on login; contains UserId, Email, Role, Expiry (60 min)
Validated by Ocelot Gateway using shared secret вҖ” services trust Gateway headers
Google OAuth: exchange Google ID token вҶ’ verify вҶ’ create/fetch AppUser вҶ’ issue own JWT
Refresh token: stored in client localStorage вҖ” stateless on the server, no DB table

3. RabbitMQ вҖ” Event-Driven Choreography
Services never call each other directly. Instead, each service publishes domain events to RabbitMQ queues; interested services subscribe and react independently. This keeps services fully decoupled вҖ” a consumer going down does not affect the publisher, and messages are retried automatically when the consumer recovers.

5. IMemoryCache Usage
Registered as a singleton via AddMemoryCache(). All keys follow a consistent naming convention. Cache is always invalidated on write вҖ” never serve stale data after a mutation.

6. Mega.nz Document Upload Flow
Customer uploads file via Claims Service POST /documents
MegaNzService streams file to Mega.nz using the MegaApiClient SDK
Returns a MegaNzFileId and share URL stored in ClaimDocuments table
Admin views documents via the share URL during claim review
On claim rejection/close, documents are retained for audit; deletion is admin-only

вҡ   Three Global Design Rules
1. GUIDs as PKs on all main entity tables (Users, Policies, Claims, etc.). Lookup/junction tables (Roles, UserRoles, InsuranceTypes, etc.) use INT IDENTITY вҖ” they are small, static, and never cross service boundaries.
2. No RefreshToken table вҖ” JWT is issued on login; the token is stored in the client's localStorage. Stateless on the server side.
3. Cross-service columns are labelled REF (no FK) вҖ” plain GUID with no enforced constraint. Consistency via RabbitMQ events.

Topic | Where It's Applied
Global Exception Handling | ExceptionMiddleware in SmartSure.Shared вҖ” one impl, 4 services register it; returns ProblemDetails
EF Core Code-First | Each service: Models вҶ’ DbContext вҶ’ Migrations вҶ’ SSMS verify
Web API Design | REST conventions, versioning-ready, DTO separation
JWT Auth & Encryption | Auth Service issues; Ocelot validates; BCrypt for passwords
RabbitMQ + Choreography | Event bus for decoupled async flows вҖ” policy activated вҶ’ email; claim submitted вҶ’ audit log; claim status changed вҶ’ email + status update. No central orchestrator; each service reacts to events independently
Swagger 6.6.2 | Per-service, with JWT Bearer security definition
Serilog | All services; JSON structured logs + rolling file sink
Docker | RabbitMQ container via docker-compose; services run locally
Unit Testing | NUnit + Moq; вүҘ90% coverage; no real DB in tests
Ocelot + Polly | Gateway routing; retry (x3) + circuit breaker per route
Repository Pattern | Controller вҶ’ Service вҶ’ IRepository вҶ’ Repository вҶ’ DbContext
Dependency Injection | All services/repos scoped; DbContext scoped; MemoryCache singleton
IMemoryCache | Products, dashboard stats, user roles вҖ” 2вҖ“10 min TTL
SMTP | MailKit; triggered by RabbitMQ consumer in Auth Service
Google OAuth | Auth Service validates Google ID token; maps to AppUser
Mega.nz | Claims Service; MegaApiClient SDK; stores handle + URL in DB
