A. Application Overview
SmartSure is an enterprise-grade, microservices-based insurance management platform that digitizes the complete insurance lifecycle. It enables customers to register, purchase policies, upload documents, and file claims through guided workflows. Admins manage products, verify claims, and generate operational reports.

B. High-Level Design (HLD)
System Architecture Diagram
All Angular UI traffic flows exclusively through the Ocelot API Gateway. The gateway validates JWT, enforces rate-limiting, and routes to the appropriate microservice. Services communicate asynchronously via RabbitMQ events. Each service owns its isolated SQL Server database.

Event-Driven Choreography Flow — Policy Purchase & Notifications
Services communicate purely through RabbitMQ events — no service calls another directly. Each service publishes an event when something meaningful happens; interested services listen and react independently. This is choreography: decoupled, asynchronous, and resilient. There is no central orchestrator coordinating the flow.

Key Design Decisions
Repository Pattern: Controller → Service → Repository → DbContext in every microservice
Dependency Injection: All services, repositories, and infrastructure registered via IServiceCollection
Global Exception Handling: Custom middleware catches all unhandled exceptions, logs via Serilog, returns RFC 7807 ProblemDetails
IMemoryCache: Applied on policy types listing, insurance product catalog, and user role lookups (5-min TTL)
Polly in Ocelot: 3 retries with exponential backoff + circuit breaker (5 failures → 30s open)
Serilog: Structured JSON logging to console + rolling file sink; includes CorrelationId per request

E. Database Design (SQL Server + EF Core Code-First)
Each microservice owns its isolated SQL Server database with its own DbContext and migrations. Cross-service references are stored as plain GUID columns — NO foreign key constraints cross service boundaries. Data consistency is maintained via RabbitMQ events.
⚠  Three Global Design Rules
1. GUIDs as PKs on all main entity tables (Users, Policies, Claims, etc.). Lookup/junction tables (Roles, UserRoles, InsuranceTypes, etc.) use INT IDENTITY — they are small, static, and never cross service boundaries.
2. No RefreshToken table — JWT is issued on login; the token is stored in the client's localStorage. Stateless on the server side.
3. Cross-service columns are labelled REF (no FK) — plain GUID with no enforced constraint. Consistency via RabbitMQ events.

F. Cross-Cutting Implementation Concepts
1. Global Exception Middleware (SmartSure.Shared)
ExceptionMiddleware lives in SmartSure.Shared — written once, used by all 4 services via project reference. Each service's Program.cs registers it with a single line. Custom exception types also live in Shared so any service can throw them and the middleware catches them uniformly.
// SmartSure.Shared/Middleware/ExceptionMiddleware.cs
using SmartSure.Shared.Exceptions;
public class ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
{
  public async Task InvokeAsync(HttpContext ctx) {
    try { await next(ctx); }
    catch (NotFoundException ex)      { await Write(ctx, 404, ex.Message); }
    catch (ValidationException ex)    { await Write(ctx, 400, ex.Message); }
    catch (UnauthorizedException ex)  { await Write(ctx, 401, ex.Message); }
    catch (Exception ex) {
      logger.LogError(ex, "Unhandled exception [{CorrelationId}]", ctx.TraceIdentifier);
      await Write(ctx, 500, "An unexpected error occurred");
    }
  }
  // Returns RFC 7807 ProblemDetails — never raw stack traces
}
// SmartSure.Shared/Exceptions/NotFoundException.cs
public class NotFoundException(string message) : Exception(message);
// SmartSure.Shared/Exceptions/ValidationException.cs
public class ValidationException(string message) : Exception(message);
// Each service Program.cs — one line, no duplication
using SmartSure.Shared.Middleware;
app.UseMiddleware<ExceptionMiddleware>();

2. JWT Configuration
Issued by Auth Service on login; contains UserId, Email, Role, Expiry (60 min)
Validated by Ocelot Gateway using shared secret — services trust Gateway headers
Google OAuth: exchange Google ID token → verify → create/fetch AppUser → issue own JWT
Refresh token: stored in client localStorage — stateless on the server, no DB table
3. RabbitMQ — Event-Driven Choreography
Services never call each other directly. Instead, each service publishes domain events to RabbitMQ queues; interested services subscribe and react independently. This keeps services fully decoupled — a consumer going down does not affect the publisher, and messages are retried automatically when the consumer recovers.
